Buddy Desktop - Final Plan Review

Revised architecture: menu bar/tray first, TypeScript owns Buddy reuse, native shell owns windows and BLE.

Date: 2026-05-19 Status: Final plan for review Supersedes old Plan A

Executive Decision

Proceed with Tauri only under a revised split: keep upstream Buddy integration in TypeScript/Node, and keep native desktop duties in Tauri/Rust or a small native helper. The default UX is a menu bar/tray app with a compact popup, so Buddy creates zero screen clutter when idle.

UX Direction

Default Surface

Buddy lives in the macOS menu bar, or the system tray on Windows/Linux. Clicking the icon opens a compact status popup.

Idle Behavior

No persistent desktop window by default. The app should feel present but quiet until something needs attention.

OpenHuman Inspiration

Borrow the polished native-app feel and companion patterns from OpenHuman, but do not copy source code or assets.

Optional Mode

A floating mascot window can be added later for users who want Buddy visible on the desktop.

Target Architecture

React UI
  -> menu bar/tray status popup
  -> Buddy mascot, stats, character display, permission overlay

Tauri native shell
  -> app lifecycle, menu bar/tray icon, popup, optional transparent mascot window
  -> BLE peripheral integration or IPC to Swift/CoreBluetooth helper

TypeScript Buddy bridge
  -> launches upstream Buddy
  -> speaks MCP stdio
  -> normalizes Buddy state
  -> exposes get_state, call_tool, subscribe_state

Reference repos
  -> buddy/ unchanged
  -> claude-desktop-buddy/ unchanged
  -> openhuman inspiration only, no code copy

Confirmed BLE Requirements

Advertisement

Act as a BLE peripheral. Device name must start with Claude. Advertise Nordic UART Service.

UUIDs

Service: 6e400001-b5a3-f393-e0a9-e50e24dcca9e

RX: 6e400002-b5a3-f393-e0a9-e50e24dcca9e

TX: 6e400003-b5a3-f393-e0a9-e50e24dcca9e

Wire Format

UTF-8 JSON, one object per line, newline terminated. Both sides reassemble fragmented BLE packets until \n.

Permissions

When a prompt arrives, send {"cmd":"permission","id":"...","decision":"once"} or deny.

Implementation Phases

0 Repo And Protocol Grounding

Keep buddy/ and claude-desktop-buddy/ as read-only references. Do not copy internals into the app.

1 Buddy Bridge Proof Of Life

Create buddy-desktop/bridge. Launch upstream Buddy, perform MCP handshake, call buddy_status, and return normalized JSON. Parser, if needed, lives only in TypeScript.

2 Minimal Menu Bar / Tray App

Scaffold Tauri + React. Tauri launches the TS bridge, creates a menu bar/tray icon, and opens a compact status popup on click.

3 Optional Floating Window

Add a user-controlled floating mascot mode only after the menu bar/tray flow works. Add macOS NSPanel only if the simpler frameless window is insufficient.

4 BLE Peripheral Prototype

Standalone BLE test: advertise Claude-XXXX, expose NUS RX/TX, receive heartbeat JSON, send status ack, and test permission response. Decide Rust btleplug vs Swift/CoreBluetooth helper here.

5 BLE App Integration

Merge Claude session state with Buddy state. Add approve/deny overlay and animation precedence.

6 Character Packs

Start with local GIF packs from ~/.buddy/characters. Defer BLE folder push until local packs are working.

First Execution Slice

  1. Scaffold buddy-desktop.
  2. Add TypeScript Buddy bridge.
  3. Launch upstream Buddy unchanged.
  4. Call buddy_status.
  5. Render normalized Buddy state in a menu bar/tray popup.
  6. Add tests around the Buddy state mapper.

Exit criteria: a menu bar/tray icon appears; clicking it shows Buddy name, level, XP, species, and stats; upstream Buddy remains unchanged; no Rust MCP parser exists.

Decision Gates

  • Do not start BLE integration until the standalone BLE peripheral prototype connects to Claude Desktop.
  • Do not implement NSPanel until the menu bar/tray popup and simpler floating window work.
  • Do not implement BLE folder push until local character packs work.
  • Do not add Buddy behavior to Rust; route Buddy tools through the TypeScript bridge.
