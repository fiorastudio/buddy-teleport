Problem Statement

fiorastudio/buddy lives entirely in the terminal — it only reacts when an AI client explicitly calls a tool. anthropics/claude-desktop-buddy gives Claude Desktop a hardware companion (ESP32) via BLE, but requires physical hardware.

This design brings both systems into a single native desktop mascot window built on tinyhumansai/openhuman (Tauri), so buddy's virtual pet is always visible and reactive to real Claude activity — no hardware required.

Chosen Approach Plan A: MCP Sidecar Plan B: BLE

Sidecar Binary: pkg approach

buddy's compiled Node.js output is bundled into a self-contained binary using pkg. Chosen over bun build --compile because buddy has native SQLite deps (better-sqlite3).

npx pkg dist/server/index.js \
  --target node18-macos-arm64 \
  --output binaries/buddy-server-aarch64-apple-darwin

claude-desktop-buddy Compatibility

The Tauri app acts as a software claude-desktop-buddy — it advertises the same Nordic UART Service BLE UUID (6e400001-b5a3-f393-e0a9-e50e24dcca9e) so Claude Desktop connects to it exactly as it would to the physical ESP32.

Character packs use the same JSON manifest + GIF format:

{ "name": "buddy-penguin",
  "colors": { "body": "#FFFFFF", "bg": "#000000" },
  "states": { "sleep": "sleep.gif", "idle": ["idle_0.gif","idle_1.gif"], ... } }

Packs install by drag-and-drop onto the mascot window or via Tauri command. Same assets work on ESP32 hardware and the desktop app.

Error Handling

FailureResponse
Sidecar crashExponential backoff restart (1→2→4→max 30s), emit buddy-offline
3 consecutive MCP errorsTrigger sidecar restart
BLE unavailable (Linux/Windows)Stub out gracefully, mascot runs on MCP data only
Character pack invalidFall back to ASCII art, log warning, no crash

Out of Scope — Plan A (deferred to Plan B)

  • BLE Nordic UART peripheral (claude-desktop-buddy session data)
  • Permission approve/deny overlay (ApproveOverlay.tsx)
  • GIF character pack support in CharacterDisplay.tsx
  • Windows/Linux BLE support
